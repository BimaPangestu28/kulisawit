## 1. Goals & Non-Goals

### Goals
- Full lahan lifecycle drivable from the browser: **create → edit → move → dispatch → watch SSE log**.
- Lahan detail drawer (shadcn `Sheet`, slides in from the right) becomes the central interaction surface for everything-after-create.
- Optimistic UI updates for column moves; pessimistic (with explicit loading state) for creates and text edits.
- Zustand debuts as the client-state store for shared UI state (selected task ID, drawer open/close, expanded attempt ID).
- Demo-able end state: click "+ Add lahan" in a column → card appears → click it → drawer opens → edit description, change column to "Doing", set agent="mock"+batch=2, click "Plant tandan" → 2 attempts appear → click one → live SSE log streams to terminal status.

### Non-Goals (3.2.2 specifically)
- **No DnD kanban.** Move is a Select dropdown in the detail drawer. (DnD lands in 3.2.3.)
- **No multi-attempt parallel log viewers.** One attempt log expanded at a time. (Tabs / split-view in 3.2.3.)
- **No inline edit ("click title to edit in place").** All edits go through the drawer. (3.2.3.)
- **No `rust-embed` wiring.** UI still served by `vite dev`. (3.2.3.)
- **No tags / linked_files editor in the detail drawer.** They're view-only there in 3.2.2; full editing arrives later.
- **No lahan delete.** Intentional YAGNI — add when there's actual demand.
- **No `GET /api/agents` endpoint.** Agent picker hardcodes `["mock"]` since the registry only contains `mock`. (Real adapters land in Phase 4 with the endpoint.)
- **No light theme.** Dark only, still locked.
- **No Playwright E2E, no CI wiring.** (Phase 5+.)

## 2. Top-Level Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Sub-phase scope | All five interactive areas (create, move, edit, dispatch, SSE) in one sub-phase | User-approved. Splitting into 3.2.2a (CRUD) + 3.2.2b (dispatch) leaves 3.2.2a feeling artificial — the unique value is "create + dispatch + watch", which only makes sense end-to-end. |
| Mutation API shape | Single `PATCH /api/tasks/:id` with partial body (`{title?, description?, column_id?}`) | Idiomatic REST. One handler delegates to existing `task::update_text` and/or `task::move_to_column`. Easier to extend later than separate endpoints. |
| Detail UI pattern | Right side panel (shadcn `Sheet`) | Preserves kanban context; switching between lahan stays one click; mobile-friendly via full-width sheet. |
| Optimism strategy | Optimistic for moves, pessimistic for create/edit | Move is frequent + reversible — must feel instant. Create/edit have explicit save buttons → users expect a loading state. |
| Client state mgmt | Zustand store (`useUiStore`) | PRD §5.10 already plans Zustand; 3.2.2 is the natural debut moment. Keeps server (TanStack Query) and client (Zustand) state cleanly separated. |
| Create UX | Inline "+ Add lahan" button per column → expands to title input + save | Trello-like familiar pattern. Title-only minimum; further fields edited via drawer after creation. |
| SSE viewer placement | List of attempts in detail drawer; click a row to expand inline log viewer below | Matches "single buah at a time" spec scope. No tab UI needed (defer to 3.2.3). |
| Agent picker | Hardcoded `["mock"]` in UI for 3.2.2 | Only `mock` registered. Adding `GET /api/agents` deferred to Phase 4 alongside real adapter additions. |
| Batch input bounds | `min=1, max=10` | Sensible cap for parallel agents on a dev machine. Adjust if PRD/usage demands. |

## 3. Backend Additions

All changes inside `crates/kulisawit-server`. **No new SQL queries** — db functions already exist.

### 3.1 `PATCH /api/tasks/:id`

**File:** `crates/kulisawit-server/src/routes/tasks.rs` (modify route registration + add handler).

**Wire DTO addition (`crates/kulisawit-server/src/wire.rs`):**

```rust
#[derive(Debug, Deserialize)]
pub struct UpdateTaskRequest {
    #[serde(default)]
    pub title: Option<String>,
    #[serde(default)]
    pub description: Option<String>,
    #[serde(default)]
    pub column_id: Option<ColumnId>,
}
```

**Handler logic:**
1. Fetch task by id (404 if missing).
2. If `req.title.is_some() || req.description.is_some()`: call `task::update_text(pool, &id, title, description)`. If only one is set, fall back to the existing value for the other (re-fetch task or pass through).
3. If `req.column_id.is_some()`: call `task::move_to_column(pool, &id, &column_id)`.
4. Re-fetch updated task and return as `TaskResponse`.

**Edge cases:**
- Empty body (no fields set): return 400 `InvalidInput("at least one field required")`.
- `column_id` references non-existent column: surfaced as DB FK error → map to 400.

**Tests (in `tests/tasks.rs`):**
- `patch_task_with_title_only_updates_title`
- `patch_task_with_column_only_moves_task`
- `patch_task_with_combined_fields_updates_both`
- `patch_unknown_task_returns_404`
- `patch_task_with_empty_body_returns_400`

## 5. Frontend Components

### 5.1 `useUiStore` (Zustand)

```typescript
import { create } from "zustand";

interface UiState {
  selectedTaskId: string | null;
  isDetailOpen: boolean;
  expandedAttemptId: string | null;
  openDetail: (taskId: string) => void;
  closeDetail: () => void;
  expandAttempt: (id: string | null) => void;
}

export const useUiStore = create<UiState>((set) => ({
  selectedTaskId: null,
  isDetailOpen: false,
  expandedAttemptId: null,
  openDetail: (id) => set({ selectedTaskId: id, isDetailOpen: true, expandedAttemptId: null }),
  closeDetail: () => set({ isDetailOpen: false, expandedAttemptId: null }),
  expandAttempt: (id) => set({ expandedAttemptId: id }),
}));
```

Selectors used at consumer level (e.g. `useUiStore((s) => s.selectedTaskId)`) keep re-renders narrow.

### 5.2 `LahanDetailSheet`

Root-level shadcn `<Sheet>` rendered once in `App.tsx`. Reads `selectedTaskId` + `isDetailOpen` from store. When open: fetches task via `useTask(selectedTaskId)`. Sheet content (vertically stacked, scrollable):
- Header: task title, close button.
- `<LahanInfoSection task={task} />`
- `<Separator />`
- `<LahanDispatchSection task={task} />`
- `<Separator />`
- `<LahanAttemptsSection task={task} />`

`onOpenChange={closeDetail}` ensures clicking outside / pressing Esc closes the sheet.

### 5.3 `LahanInfoSection`

Form region with controlled inputs:
- `<Input>` for title (initial value from `task.title`)
- `<Textarea>` for description (initial from `task.description ?? ""`)
- `<Select>` for column (options from current `useBoard` cache; initial value `task.column_id`)
- `<Button>` "Save" — disabled unless title or description differ from initial values.

**Behavior:**
- Title / description: changes are local state. Save button calls `useUpdateTask` with only the changed fields. Pessimistic — button shows spinner, disabled until response.
- Column dropdown: changes fire `useUpdateTask({column_id})` **immediately** (no save click). Optimistic — kanban updates instantly via TanStack cache mutation.

Errors render inline (red text under the field group) on mutation error.

### 5.4 `LahanDispatchSection`

```
[ Agent: <Select with "mock" only> ]
[ Batch: <Input type=number min=1 max=10 default=1> ]
[ Plant tandan ] (Button)
```

On click: `useDispatchTask({ id: task.id, body: { agent, batch } })`. While pending: button shows spinner + disabled. On success: invalidate `["task-attempts", task.id]` so the attempts section re-fetches.

### 5.5 `LahanAttemptsSection`

Wraps `useTaskAttempts(task.id)`.
- **Loading:** small skeleton row.
- **Empty:** muted text "No attempts yet — plant a tandan above."
- **Loaded:** ordered list of attempt rows. Each row:
  - Status badge (color-coded: queued=secondary, running=blue, completed=green, failed=destructive, cancelled=muted)
  - Agent name + prompt_variant (if present)
  - Started timestamp (relative: "2m ago")
  - Click row → toggles expanded inline `<AttemptLogViewer attemptId={a.id} />` below it.
  - Only one expanded at a time (managed via `expandedAttemptId` in store).

### 5.6 `AttemptLogViewer`

Wraps `useAttemptEvents(attemptId)`. Renders:
- Loading shim "Connecting…" while no events received.
- Once events arrive: monospace pre-block, newest at bottom, max-height with scroll. Auto-scroll-to-bottom logic only if user is already at bottom (don't yank scroll if user is reading mid-stream).
- Each line: `[hh:mm:ss] {kind}: {message}` (status events styled distinctly).

### 5.7 `AddTaskInline`

Two states managed locally:
- **Idle:** Button "+ Add lahan" (subtle, full-width within column footer).
- **Editing:** Input (autofocused, placeholder "Lahan title") + Save button + Cancel button. Esc collapses; Enter submits.

Submit calls `useCreateTask({ project_id, column_id, title })`. While pending: both buttons disabled, spinner on Save. On success: collapse to idle + clear input. On error: show inline error below input, keep editing state.

### 5.8 Modifications to existing components

- **`KanbanColumn.tsx`** — render `<AddTaskInline projectId={...} columnId={column.id} />` after the task list (always visible, even when column has tasks). Pass `projectId` from parent.
- **`TaskCard.tsx`** — wrap the existing card content in `<button onClick={() => openDetail(task.id)} className="text-left w-full block">`. Add `cursor-pointer hover:bg-accent` to the existing Card. The button must be a real `<button>` for keyboard accessibility (Enter/Space activate).
- **`App.tsx`** — add `<LahanDetailSheet />` once at root, after the `<header>` and before `<main>` (Sheet portals itself, position doesn't visually matter, but keep it close to siblings).

## 6. Data Flow + Mutations

### 6.1 Mutation patterns

**`useCreateTask` (pessimistic):**
```typescript
useMutation({
  mutationFn: (req: CreateTaskRequest) => api.post<Task>("/tasks", req),
  onSuccess: (created) => {
    queryClient.invalidateQueries({ queryKey: ["board", created.project_id] });
  },
});
```

**`useUpdateTask` (optimistic for `column_id`, pessimistic for text):**
```typescript
useMutation({
  mutationFn: ({ id, body }: { id: string; body: UpdateTaskRequest }) =>
    api.patch<Task>(`/tasks/${id}`, body),
  onMutate: async ({ id, body }) => {
    if (body.column_id === undefined) return; // text-only edits skip optimism
    await queryClient.cancelQueries({ queryKey: ["board"] });
    const snapshot = queryClient.getQueriesData({ queryKey: ["board"] });
    queryClient.setQueriesData<BoardResponse>({ queryKey: ["board"] }, (old) => {
      if (!old) return old;
      const allTasks = old.columns.flatMap((c) => c.tasks);
      const moved = allTasks.find((t) => t.id === id);
      if (!moved) return old;
      const updated = { ...moved, column_id: body.column_id! };
      return {
        ...old,
        columns: old.columns.map((c) => ({
          ...c,
          tasks:
            c.id === body.column_id
              ? [...c.tasks.filter((t) => t.id !== id), updated]
              : c.tasks.filter((t) => t.id !== id),
        })),
      };
    });
    return { snapshot };
  },
  onError: (_err, _vars, ctx) => {
    if (ctx?.snapshot) {
      ctx.snapshot.forEach(([key, data]) => queryClient.setQueryData(key, data));
    }
  },
  onSettled: () => queryClient.invalidateQueries({ queryKey: ["board"] }),
});
```

**`useDispatchTask` (pessimistic):**
```typescript
useMutation({
  mutationFn: ({ id, body }: { id: string; body: DispatchRequest }) =>
    api.post<DispatchResponse>(`/tasks/${id}/dispatch`, body),
  onSuccess: (_resp, vars) => {
    queryClient.invalidateQueries({ queryKey: ["task-attempts", vars.id] });
  },
});
```

### 6.2 `lib/api.ts` extension

Add `patch` helper:
```typescript
export const api = {
  get: <T>(path: string) => request<T>(path),
  post: <T>(path: string, body: unknown) =>

};
```

## 7. SSE Flow

`useAttemptEvents` is intentionally NOT a TanStack Query — `EventSource` requires lifecycle management that doesn't fit Query's poll/cache model.

```typescript
const TERMINAL: AttemptStatus[] = ["completed", "failed", "cancelled"];

export function useAttemptEvents(attemptId: string | null) {
  const queryClient = useQueryClient();
  const [events, setEvents] = useState<AgentEvent[]>([]);

  useEffect(() => {
    if (!attemptId) return;
    setEvents([]);
    const es = new EventSource(`/api/attempts/${attemptId}/events`);

    es.onmessage = (msg) => {
      const event = JSON.parse(msg.data) as AgentEvent;
      setEvents((prev) => [...prev, event]);
      if (event.type === "status" && TERMINAL.includes(event.status)) {
        es.close();
        queryClient.invalidateQueries({ queryKey: ["attempt", attemptId] });
        queryClient.invalidateQueries({ queryKey: ["task-attempts"] });
        queryClient.invalidateQueries({ queryKey: ["board"] });
      }
    };

    es.onerror = () => {
      // Auto-reconnect or close — for 3.2.2 we just close; user clicks again to retry.
      es.close();
    };

    return () => es.close();
  }, [attemptId, queryClient]);

  return events;
}
```

Component renders `events` directly. Stream close on terminal status invalidates relevant caches so the kanban / attempts list reflect final state.
