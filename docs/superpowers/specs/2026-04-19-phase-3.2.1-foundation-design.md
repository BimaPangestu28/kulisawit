# Phase 3.2.1 — Foundation (Read-Only Kanban) Design Spec

**Date:** 2026-04-19
**Status:** Approved (brainstorming complete, ready for implementation plan)
**Phase:** 3.2.1 of the 3.2.x sub-phases (Foundation → Interactivity → Polish)
**Prerequisite:** Phase 3.1 complete (HEAD `74a9a7b`+); README shipped; backend has 7 endpoints under `/api`.

---

## 1. Goals & Non-Goals

### Goals
- Browser shows a real kanban board, populated with real data, fetched from `kulisawit serve`.
- Frontend scaffold is complete and idiomatic (Vite + React 18 + TypeScript + Tailwind + shadcn/ui + TanStack Query + Vitest + MSW).
- Dark theme, Poppins font, minimal emoji — locked at scaffold-time so subsequent sub-phases don't have to revisit style decisions.
- Backend grows just enough to make the kanban renderable: auto-seed columns on project creation, list projects, fetch a project's full board.
- Demo-able end state: `kulisawit serve --port 7700` plus `cd ui && npm run dev` (port 5173). Open `http://localhost:5173`, see project switcher dropdown, pick a project, see five-column kanban with task cards (or empty-column placeholders).

### Non-Goals (3.2.1 specifically)
- **No interactivity.** No create lahan, no move, no dispatch, no SSE viewer. (All deferred to 3.2.2.)
- **No DnD.** No multi-attempt tabs. No inline edit. (All deferred to 3.2.3.)
- **No `rust-embed` wiring.** UI not yet baked into the binary. (Deferred to 3.2.3.)
- **No Playwright E2E.** Vitest + MSW only.
- **No CI wiring.** Local-only testing.
- **No light theme / theme toggle.** Dark only, hard-coded.
- **No real kuli adapters in any UI dropdown.** (Mock agent exists in backend; agent picker first appears in 3.2.2 and lists whatever the registry has.)

## 2. Top-Level Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Sub-phase decomposition | Three sub-phases: 3.2.1 Foundation, 3.2.2 Interactivity, 3.2.3 Polish | "Polished UX" full scope is ~3-4 weeks of work; one spec per sub-phase keeps each cycle reviewable. |
| 3.2.1 layering | Backend + frontend in lockstep (Approach A) | Sub-phase already small; backend additions are tightly coupled to frontend's render needs; splitting adds tracking overhead with no isolation benefit. |
| Style system | shadcn/ui (default style, `zinc` base color) | User-set; matches the technical/clean aesthetic the project is going for. |
| Theme | Dark only, `<html class="dark">` hard-coded | User-set. Light theme not on the roadmap. |
| Font | Poppins (300–700, via Google Fonts `<link>` in `index.html`) | User-set. |
| Emoji policy | Zero in components; ✅ ⚠ permitted sparingly in error/status banners | User-set. Matches the README's discipline. |
| Backend column seeding | Auto-seed 5 default columns inside `POST /api/projects` | Simplest UX. Closes the trap where a fresh project couldn't accept tasks because no columns existed. |
| Tasks list shape | Grouped by column inside one `GET /api/projects/:id/board` endpoint | Single round-trip per kanban render. Matches the data the UI actually needs. |
| Dev workflow | Vite dev server (`:5173`) with `/api` proxy to backend (`:7700`) | Fast iteration. `rust-embed` deferred to 3.2.3. |
| TS types | Hand-rolled in `src/types/api.ts` | Surface is small (~7 types). Keeps build simple; drift mitigated by a comment in `wire.rs`. |
| Frontend tests | Vitest + `@testing-library/react` + MSW | Component coverage for ProjectSwitcher / KanbanBoard / TaskCard. No browser E2E in 3.2.1. |
| State management | TanStack Query for server state; `localStorage` + `useState` for `activeProjectId`; no Zustand yet | YAGNI. Zustand introduced in 3.2.2 when interaction state grows. |
| Naming convention | English component / prop / hook names; Indonesian Kulisawit terms in user-facing strings only | Consistent with the project-wide rule from PRD §0 + the English-generic-naming feedback memory. |
| Listing endpoint placement | `GET /api/projects` (list all) added; flat `GET /api/projects/:id/tasks` NOT added (board endpoint covers UI need) | Smallest surface that solves the actual problem. |

## 3. Backend Additions

All changes land inside `crates/kulisawit-server` (and one helper inside `crates/kulisawit-db` if a list function isn't already present). No migrations needed — the schema already supports everything.

### 3.1 Modify `POST /api/projects` to auto-seed columns

**File:** `crates/kulisawit-server/src/routes/projects.rs`

After `project::create` succeeds, call `kulisawit_db::columns::seed_defaults(pool, &id)`. The function already exists (`crates/kulisawit-db/src/columns.rs:23`) and is exercised by every existing test that needs columns; production just hadn't called it.

**Wire DTO change (`crates/kulisawit-server/src/wire.rs`):**

```rust
#[derive(Debug, Serialize)]
pub struct ProjectResponse {
    pub id: ProjectId,
    pub name: String,
    pub repo_path: String,
    pub created_at: i64,
    // NEW. Always present in JSON (no skip_serializing_if). Filled with the 5
    // freshly-seeded column IDs on the POST /api/projects response. Always an
    // empty Vec on GET /api/projects (list) responses; clients that need the
    // column IDs after listing should call GET /api/projects/:id/board.
    #[serde(default)]
    pub column_ids: Vec<ColumnId>,
}
```

**Updated handler shape:**

```rust
async fn create(
    State(state): State<AppState>,
    Json(req): Json<NewProjectRequest>,
) -> ServerResult<Json<ProjectResponse>> {
    let id = project::create(state.orch.pool(), NewProject {
        name: req.name.clone(),
        repo_path: req.repo_path.clone(),
    }).await?;
    let column_ids = columns::seed_defaults(state.orch.pool(), &id).await?;
    let row = project::get(state.orch.pool(), &id).await?
        .ok_or_else(|| ServerError::Internal("project vanished after insert".into()))?;
    let mut resp: ProjectResponse = row.into();
    resp.column_ids = column_ids;
    Ok(Json(resp))
}
```

**Test additions / changes:**
- Update `post_projects_with_valid_body_returns_200_and_inserts` to assert response includes `column_ids` of length 5, and that those IDs are present in the `columns` table for the project, in default order (`Backlog`, `Todo`, `Doing`, `Review`, `Done`).

### 3.2 Add `GET /api/projects` (list all)

**File:** `crates/kulisawit-server/src/routes/projects.rs` (add a route + handler)
**Helper:** `crates/kulisawit-db/src/project.rs` already has `list(pool)` (verified at HEAD `74a9a7b`, line 63). Reuse it.

**Response:** `Vec<ProjectResponse>`. Per the wire DTO note in §3.1, `column_ids` is always serialized but always empty on this endpoint. Clients that need column IDs after listing call `GET /api/projects/:id/board`.

**Tests:**
- `get_projects_returns_empty_array_when_none`

### 3.3 Add `GET /api/projects/:id/board`

**File (new):** `crates/kulisawit-server/src/routes/board.rs`
**Wire DTOs (add to `wire.rs`):**

```rust
#[derive(Debug, Serialize)]
pub struct BoardResponse {
    pub project: ProjectResponse,
    pub columns: Vec<BoardColumn>,
}

#[derive(Debug, Serialize)]
pub struct BoardColumn {
    pub id: ColumnId,
    pub name: String,
    pub position: i64,
    pub tasks: Vec<TaskResponse>,
}
```

**Handler logic:**
1. Fetch project by id (404 if missing).
2. Fetch columns via `columns::list_for_project(pool, &id)` — returns ordered by position ASC.
3. Fetch all tasks for the project (add a small helper if needed: `task::list_for_project(pool, &project_id) -> DbResult<Vec<Task>>`; currently the task repo has `list_for_column` only — verify and add if absent).
4. Group tasks by `column_id` in Rust (HashMap or single-pass with column-position map).
5. For each column in position order, attach its task list (also ordered by `task.position`).
6. Serialize as `BoardResponse`.

**Mounting:** add to `routes/mod.rs` router composition.

**Tests (`crates/kulisawit-server/tests/board.rs`):**
- `get_board_returns_5_empty_columns_for_fresh_project` — create project (which now auto-seeds), GET board, assert 5 columns, each with `tasks: []`, names/positions match defaults.
- `get_board_groups_tasks_into_correct_columns` — create project, insert tasks across columns 1 and 3 with varying positions, GET board, assert grouping correct and within-column order respects `position`.
- `get_board_returns_404_for_unknown_project` — assert NOT_FOUND.

## 5. Frontend Components

Five custom components plus five lazily-installed shadcn primitives. Read-only — no event handlers attached in 3.2.1 except localStorage write in `ProjectSwitcher.onValueChange`.

### 5.1 `App.tsx`

Layout shell. Renders header + body. Header is sticky top, contains app title `Kulisawit` (left) and `<ProjectSwitcher />` (right). Body renders one of:
- `<KanbanBoard projectId={activeProjectId} />` if `activeProjectId` is set,
- `<EmptyState>` with copy "Pick a project to view its board" if projects exist but none selected,
- `<EmptyState>` with copy "No projects yet. Create one with `curl -X POST /api/projects` until 3.2.2 lands the create form." if list is empty.

Manages `activeProjectId` state via `useState` + a `useEffect` that reads `localStorage["kulisawit.activeProject"]` on mount and writes whenever it changes.

### 5.2 `ProjectSwitcher.tsx`

shadcn `Select`. Wraps `useProjects()`.
- **Loading:** `<Skeleton className="h-9 w-48" />`
- **Error:** small destructive Card with "Failed to load projects" + Retry button (calls `queryClient.invalidateQueries({queryKey: ['projects']})`)
- **Empty:** disabled Select with placeholder "No projects"
- **Loaded:** shadcn Select with one item per project (label = project name, value = project id). Triggers `onSelect(id)` → updates parent state.

Auto-selects the first project on mount if `activeProjectId` is null AND list is non-empty.

### 5.3 `KanbanBoard.tsx`

Wraps `useBoard(projectId)`. Props: `{ projectId: string }`.
- **Loading:** 5 column-shaped Skeletons in a row.
- **Error:** destructive Card "Failed to load board" + Retry.
- **Loaded:** CSS grid with `grid-template-columns: repeat(5, minmax(280px, 1fr))`, horizontal overflow on small viewports. Renders `<KanbanColumn>` per column from `board.columns` in order.

### 5.4 `KanbanColumn.tsx`

Props: `{ column: BoardColumn }`.
- Header row: column name (text-sm font-medium) + task count Badge (`{tasks.length}`).
- Body: vertical stack of `<TaskCard>`. Empty state inside column: muted-foreground "No tasks" hint, centered.

### 5.5 `TaskCard.tsx`

Props: `{ task: Task }`. shadcn `Card` (subtle border, hover bg shift).
- Title (always visible, font-medium).
- Description (line-clamp-2 if present, hidden if null/empty).
- Tags row: each tag rendered as shadcn `Badge variant="secondary"` (skipped entirely if `tags` array empty).
- Linked-files indicator (subtle): if `linked_files.length > 0`, show `📎 N files` count in muted-foreground tiny text. **Exception to the no-emoji rule** — `📎` is a UI affordance signal, not decoration. (If you want zero emoji at all, swap for `[N files]` text.)

### 5.6 `EmptyState.tsx`

Generic. Centered shadcn Card with title + description + optional action slot. Used for the two no-data scenarios in `App.tsx`.

### 5.7 shadcn primitives (no customization beyond install defaults)

- `button` (used in EmptyState action slot, error retry)
- `card` (TaskCard, EmptyState, error states)
- `select` (ProjectSwitcher)
- `skeleton` (loading states)
- `badge` (column count, task tags)

Install order: `npx shadcn@latest init` (uses `components.json`), then `npx shadcn@latest add button card select skeleton badge`.

## 6. Data Flow

### 6.1 Initial load sequence

```
mount App
  ├─ read localStorage["kulisawit.activeProject"] → state.activeProjectId
  ├─ render <ProjectSwitcher />
  │    └─ useProjects() → GET /api/projects (proxied to :7700)
  │         ├─ loading → Skeleton
  │         ├─ error   → Card with Retry
  │         └─ success → Select with items
  │              ├─ if activeProjectId matches a returned id → keep
  │              ├─ else if list non-empty → set activeProjectId = projects[0].id; persist
  │              └─ else                    → render <EmptyState> in body
  └─ render <KanbanBoard projectId={activeProjectId} />  (conditional)
       └─ useBoard(activeProjectId) → GET /api/projects/:id/board
            ├─ loading → 5 column Skeletons
            ├─ error   → Card with Retry
            └─ success → 5 KanbanColumn components
                 └─ TaskCard per task
```

### 6.2 Selection change

`ProjectSwitcher.onValueChange(id)` → calls parent `setActiveProjectId(id)` → `useEffect` writes to localStorage → `KanbanBoard` re-renders with new `projectId` → TanStack Query fires new `GET /api/projects/:id/board`. Old query result stays cached (default `cacheTime: 5min`); switching back is instant.

### 6.3 Error handling

All hooks throw on `!res.ok`. Errors land in TanStack Query's `error` field. UI renders a destructive shadcn Card with the error's message and a Retry button calling `queryClient.invalidateQueries(...)`.

Network down ≠ blank screen. Backend returns 4xx/5xx ≠ silent fail.

### 6.4 No mutations

`useMutation` does not appear in 3.2.1. First introduced in 3.2.2.

## 11. Demo Acceptance Criteria

A reviewer should be able to:

1. Pull `main` at the post-3.2.1 HEAD.
2. Run `cargo build --release && ./target/release/kulisawit serve --db /tmp/k.db --repo /tmp/somerepo --port 7700` (after `git init` in `/tmp/somerepo`).
3. Run `cd ui && npm install && npm run dev`.
4. Open `http://localhost:5173` in a browser.
5. See a sticky header with "Kulisawit" title and a project switcher.
6. The switcher is empty initially. Use `curl -X POST http://localhost:7700/api/projects -H 'content-type: application/json' -d '{"name":"demo","repo_path":"/tmp/somerepo"}'` to create a project.
7. Refresh the browser. The switcher now shows "demo". Select it.
8. See 5 empty columns: Backlog, Todo, Doing, Review, Done.
9. Insert a task directly via SQL or `POST /api/tasks` (with `column_id` from step 6's response, since `POST /api/tasks` is already implemented from Phase 3.1). Refresh.
10. The task card appears in the correct column.

**Test gate:**
- `cargo test --workspace` returns ≥115 passed.
- `cd ui && npm test` returns all green (≥8 tests).
- `cargo clippy --workspace --all-targets -- -D warnings` clean.
