# Phase 3.3.1 — Sortir (Verification Runner) Design Spec

**Date:** 2026-04-25
**Status:** Approved (brainstorming complete, ready for implementation plan)
**Phase:** 3.3.1 of the 3.3.x sub-phases (**Sortir** → Panen)
**Prerequisite:** Phase 3.2.3 complete (HEAD `fe378a6` or later — single-binary UI demo working, 125 backend tests default / 127 with embed-ui + 29 frontend tests baseline).

---

## 1. Goals & Non-Goals

### Goals
- **Auto-trigger** sortir runner when an attempt reaches agent terminal `Succeeded`. Runs user-defined checks (test, lint, build, etc.) from `.kulisawit/sortir.toml` in the petak (worktree) cwd.
- **Capture aggregated output** + persist `verification_status` (pending → passed/failed/skipped) and `verification_output` in the existing `attempt` table columns (Phase 1 schema).
- **All-pass = passed; any-fail = failed.** Run all configured checks regardless (for full log); seal status as failed on first non-zero exit but keep running rest.
- **Symmetric SSE close path:** every terminal flow ends with a synthetic `Status` event whose `detail` starts with `"sortir:"`. SSE consumer waits for that prefix before closing.
- **No config file → skipped.** Agent terminal Failed/Cancelled → also skipped.
- **UI:** colored dot badge (passed/failed/pending/skipped) on attempt tab + dedicated "Verification" section in attempt panel showing duration + output text.
- **Demo-able end:** create `.kulisawit/sortir.toml` with `cargo test`, dispatch task, watch attempt: SSE shows agent done → tab badge pulses pending → sortir runs → badge turns green/red → Verification section shows aggregated check output blocks.

### Non-Goals (3.3.1 specifically)
- **No panen UI / no merge logic.** Phase 3.3.2 owns it.
- **No GUI sortir config editor.** File-only for v0.1.
- **No per-task sortir override.** Project-level only.
- **No fail-fast.** Always run all checks for the full log.
- **No streaming sortir output.** Sortir result emitted as one Status event when done; check outputs aren't live-streamed via SSE.
- **No DAG / inter-check dependencies.** Sequential execution.
- **No HTTP endpoint to manually trigger sortir.** Auto-only on agent terminal Succeeded.
- **No re-run UI.** If a user wants a re-run, dispatch a new attempt.
- **No per-check status JSON column.** Single aggregated `verification_status` + `verification_output` text.
- **No live-streaming sortir output via SSE.** Could add later via a new event variant.

## 2. Top-Level Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Sub-phase decomposition | 3.3.1 Sortir + 3.3.2 Panen (sequential) | User-approved. Sortir produces verification status which Panen consumes; clean dependency order. |
| Sortir config storage | `<repo_root>/.kulisawit/sortir.toml` (per-project file) | User-approved. Versioned with repo, human-editable, sensible default (skipped) when absent. |
| Status aggregation | All-pass = passed; any-fail = failed; run all for log | User-approved. Conservative for panen safety; full log for debugging. |
| UI surface | Dot badge on tab + dedicated section in attempt panel | User-approved. Persistent visibility from tab list, full detail on demand. |
| SSE update mechanism | Reuse existing `Status` event with `"sortir:..."` detail prefix | User-approved. Avoid new event variants; minimize schema churn. |
| SSE close-rule change | Close only on `Status` whose `detail` starts with `"sortir:"` | Required to keep stream open while sortir runs after agent terminal. |
| Sortir trigger | Auto on agent `Status::Succeeded` only; Failed/Cancelled → skipped without spawning runner | Avoids running checks against broken worktrees. Keeps SSE close path symmetric (every terminal → some `sortir:*` Status frame). |
| Sortir runner location | New module `crates/kulisawit-orchestrator/src/sortir.rs` (NOT a separate crate) | Tight coupling to attempt lifecycle + broadcaster. Crate-level extraction is YAGNI for v0.1. |
| Output cap | 64 KB per check (truncate stdout/stderr); 256 KB total (drop oldest blocks) | Bounded DB write size; avoids huge `verification_output` rows. |
| Default check timeout | 1800s hard cap, no implicit default (user must specify `timeout_secs`) | Explicit; avoids accidental zombies. |
| Toml validation errors | `set_verification(Failed, "sortir.toml invalid: <err>")` + `sortir:failed` broadcast | Clear surfacing of config bugs vs check failures. |
| Tooltip primitive | Add shadcn `tooltip` (one new primitive) | Required for `VerificationBadge` accessible label. |

### 3.1 New module `crates/kulisawit-orchestrator/src/sortir.rs`

**Public surface:**
```rust
pub async fn run_sortir(orch: Arc<Orchestrator>, attempt_id: AttemptId);
```

**Internals:**
- `struct SortirConfig { checks: Vec<Check> }` — toml::Deserialize
- `struct Check { name: String, command: Vec<String>, timeout_secs: u64 }` — validated on parse: name non-empty, command non-empty, timeout 1..=1800
- `async fn load_config(repo_root: &Path) -> Option<SortirConfig>` — reads `<repo_root>/.kulisawit/sortir.toml`; returns `None` if file absent. `Some(Err(...))` for invalid toml is mapped at the call site to `set_verification(Failed, "sortir.toml invalid: <e>")`.
  - **Refinement:** signature actually `Result<Option<SortirConfig>, ConfigError>` so the caller distinguishes "absent" from "malformed". Plan-writing locks the exact return type.
- `async fn run_check(check: &Check, cwd: &Path) -> CheckResult { name, status: Pass|Fail|Timeout, output: String, duration_ms: u64 }`
  - Spawn via `tokio::process::Command::new(&check.command[0]).args(&check.command[1..]).current_dir(cwd).stdout(Stdio::piped()).stderr(Stdio::piped()).kill_on_drop(true).spawn()`
  - `tokio::time::timeout(Duration::from_secs(check.timeout_secs), child.wait_with_output())` — on timeout, the kill_on_drop drop reaps the child.
  - Combined stdout+stderr capture; cap at 64 KB each (truncate, append `"... (truncated)"`).
- `fn aggregate(results: Vec<CheckResult>) -> (VerificationStatus, String)` — all-pass → Passed; any non-pass → Failed; format text as `"=== <name> (<status>, <ms>ms) ===\n<output>\n\n"` blocks. Total cap 256 KB by dropping oldest blocks with marker `"--- truncated <N> earlier blocks ---\n"` prepended.

### 3.2 Hook in `crates/kulisawit-orchestrator/src/dispatch.rs`

In `dispatch_one_attempt` (currently lines 162 + 286), after the existing `attempt::mark_terminal(orch.pool(), &attempt_id, terminal).await?` call:

```rust
match terminal {
    AttemptStatus::Completed => {
        // Mark verification pending so UI shows pulsing badge immediately.
        attempt::set_verification(orch.pool(), &attempt_id, VerificationStatus::Pending, Some("queued"))
            .await
            .ok(); // best-effort; don't fail the attempt if write hiccups
        let orch_clone = Arc::clone(&orch);
        let id = attempt_id.clone();
        tokio::spawn(async move {
            sortir::run_sortir(orch_clone, id).await;
        });
    }
    AttemptStatus::Failed | AttemptStatus::Cancelled => {
        attempt::set_verification(
            orch.pool(),
            &attempt_id,
            VerificationStatus::Skipped,
            Some("agent did not complete successfully"),
        )
        .await
        .ok();
        // Inline broadcast of synthetic terminal Status with sortir:skipped detail.
        let derived_run_status = match terminal {
            AttemptStatus::Failed => RunStatus::Failed,
            AttemptStatus::Cancelled => RunStatus::Cancelled,
            _ => unreachable!(),
        };
        orch.broadcaster().send(
            attempt_id.clone(),
            AgentEvent::Status {
                status: derived_run_status,
                detail: Some("sortir:skipped".into()),
            },
        );
    }
    _ => {} // Queued/Running aren't terminal; this match is exhaustive at the call site.
}
```

The `set_verification` to `Pending` is critical: when the user opens the drawer between agent-terminal and sortir-done, the UI must show the pulsing pending state, not "no data yet".

### 3.3 `run_sortir` body (high level)

```rust
pub async fn run_sortir(orch: Arc<Orchestrator>, attempt_id: AttemptId) {
    let attempt = match attempt::get(orch.pool(), &attempt_id).await {
        Ok(Some(a)) => a,
        _ => return, // attempt vanished; nothing to do
    };
    let repo_root = &orch.repo_root(); // existing accessor, see Orchestrator
    let worktree = Path::new(&attempt.worktree_path);

    let (final_status, output) = match load_config(repo_root).await {
        Ok(None) => (
            VerificationStatus::Skipped,
            "no .kulisawit/sortir.toml found".to_string(),
        ),
        Err(e) => (
            VerificationStatus::Failed,
            format!("sortir.toml invalid: {e}"),
        ),
        Ok(Some(config)) => {
            let mut blocks: Vec<String> = Vec::new();
            let mut all_pass = true;
            for check in &config.checks {
                let result = run_check(check, worktree).await;
                if !matches!(result.status, CheckOutcome::Pass) {
                    all_pass = false;
                }
                blocks.push(format_block(&result));
            }
            let status = if all_pass {
                VerificationStatus::Passed
            } else {
                VerificationStatus::Failed
            };
            (status, aggregate_blocks(blocks))
        }
    };

    let _ = attempt::set_verification(orch.pool(), &attempt_id, final_status, Some(&output)).await;

    // Derive RunStatus from agent's actual terminal (read from DB row) — already in attempt.status.
    let derived = match attempt.status {
        AttemptStatus::Completed => RunStatus::Succeeded,
        AttemptStatus::Failed => RunStatus::Failed,
        AttemptStatus::Cancelled => RunStatus::Cancelled,
        _ => RunStatus::Failed, // shouldn't happen at this point; defensive
    };
    let detail = match final_status {
        VerificationStatus::Passed => "sortir:passed",
        VerificationStatus::Failed => "sortir:failed",
        VerificationStatus::Skipped => "sortir:skipped",
        VerificationStatus::Pending => "sortir:pending", // shouldn't reach here
    };
    orch.broadcaster().send(
        attempt_id,
        AgentEvent::Status {
            status: derived,
            detail: Some(detail.to_string()),
        },
    );
}
```

Notes:
- `orch.repo_root()` — verify accessor exists; if not, add a small public method (Orchestrator already stores it from `Orchestrator::new`).
- `orch.broadcaster()` already exists (Phase 2.1.3).
- All write errors are best-effort (log via `tracing::warn`, don't unwrap).

### 5.5 `useAttemptEvents` close-rule change

`src/hooks/useAttemptEvents.ts`:

```typescript
es.onmessage = (msg: MessageEvent) => {
  try {
    const envelope = JSON.parse(msg.data) as EventEnvelope;
    setEvents((prev) => [...prev, { envelope, receivedAt: Date.now() }]);
    if (envelope.event.type === "status") {
      const detail = envelope.event.detail ?? "";
      const isSortirDone = detail.startsWith("sortir:");
      // Only close on the final sortir Status. The agent's own terminal
      // Status (detail = null) keeps the stream open so the sortir-done
      // frame can arrive.
      if (isSortirDone) {
        es.close();
        queryClient.invalidateQueries({ queryKey: ["attempt", attemptId] });
        queryClient.invalidateQueries({ queryKey: ["task-attempts"] });
        queryClient.invalidateQueries({ queryKey: ["board"] });
      }
    }
  } catch {
    // malformed frame — ignore
  }
};
```

The previous `TERMINAL_RUN_STATUSES.includes(...)` check is **removed** in favor of the `detail` prefix check. The agent terminal Status now arrives but doesn't close the stream.

## 6. Data Flow

### 6.1 Sortir lifecycle (full sequence)

```
1. dispatch_one_attempt: agent runs in petak → emits Stdout/Stderr/etc events
   → reaches RunStatus::Succeeded
   → broadcaster sends Status::Succeeded (detail: None)
   → mark_terminal(AttemptStatus::Completed) writes DB
   → set_verification(Pending, "queued") writes DB
   → tokio::spawn(sortir::run_sortir(orch, attempt_id))
   → agent task done

2. SSE consumer (useAttemptEvents):
   → receives Status::Succeeded with detail=None
   → does NOT close (new rule: close only on "sortir:" prefix)
   → renders VerificationSection in pending mode (pulsing dot)

3. sortir::run_sortir (background):
   a. Load attempt row → worktree_path
   b. load_config(repo_root):
      - Absent: set_verification(Skipped, "no .kulisawit/sortir.toml found")
        + broadcast Status with detail "sortir:skipped" → return
      - Invalid: set_verification(Failed, "sortir.toml invalid: <err>")
        + broadcast Status with detail "sortir:failed" → return
      - Valid: continue to (c)
   c. For each check sequentially:
      - tokio::process::Command in worktree cwd, capture stdout+stderr
      - tokio::time::timeout per check.timeout_secs
      - On timeout: kill_on_drop reaps; record TIMEOUT marker
      - On exit: cap output 64 KB per stream; format block
   d. Aggregate: status = passed (all OK) | failed (any non-pass)
   e. Cap total at 256 KB (drop oldest blocks; prepend truncation marker)
   f. set_verification(status, output) writes DB
   g. broadcast Status with detail "sortir:<status>"

4. SSE consumer:
   → receives Status with detail starting "sortir:"
   → closes EventSource
   → invalidates ["attempt", id], ["task-attempts"], ["board"]
   → useTaskAttempts refetches; AttemptResponse has populated verification fields
   → VerificationBadge re-renders with correct color
   → VerificationSection shows the output blocks
```

### 6.2 Agent-failed/cancelled path (no spawn)

If terminal == `Failed | Cancelled`, dispatch.rs **inline**:
1. `set_verification(Skipped, "agent did not complete successfully")` — DB write.
2. Broadcast `Status { status: <derived>, detail: Some("sortir:skipped".into()) }`.

This keeps the SSE consumer's close path symmetric: every terminal flow ends with a `sortir:*` Status event.

## 7. Configuration File Format

`<repo_root>/.kulisawit/sortir.toml`:

```toml
[[checks]]
name = "test"
command = ["cargo", "test", "--workspace"]
timeout_secs = 300

[[checks]]
name = "lint"
command = ["cargo", "clippy", "--workspace", "--", "-D", "warnings"]
timeout_secs = 120

[[checks]]
name = "fmt"
command = ["cargo", "fmt", "--", "--check"]
timeout_secs = 60
```

Validation on parse:
- `name` non-empty string
- `command` non-empty array of strings
- `timeout_secs` integer in `1..=1800`

Parse errors are mapped to `set_verification(Failed, "sortir.toml invalid: <e>")` + `sortir:failed` broadcast.

### Output format example

```
=== test (passed, 4250ms) ===
running 142 tests
test result: ok. 142 passed; 0 failed

=== lint (failed, 1820ms) ===
warning: unused variable `x`
error: aborting due to previous error

=== build (passed, 8400ms) ===
   Compiling kulisawit v0.1.0
    Finished `dev` profile in 8.3s
```

Truncation marker (if applied): `"\n--- truncated <N> earlier blocks ---\n"` prepended after dropping oldest blocks.

## 12. Demo Acceptance Criteria

A reviewer should be able to:

1. `cd ui && npm run build && cd .. && cargo build --release --features kulisawit-server/embed-ui`
2. Set up demo repo with sortir config:
   ```bash
   TMPREPO=$(mktemp -d)
   cd "$TMPREPO" && git init -q && echo init > README && git add README && git commit -m init
   mkdir -p .kulisawit
   cat > .kulisawit/sortir.toml <<'EOF'
   [[checks]]
   name = "smoke-pass"
   command = ["true"]
   timeout_secs = 5

   [[checks]]
   name = "smoke-fail"
   command = ["false"]
   timeout_secs = 5
   EOF
   ```
3. `./target/release/kulisawit serve --db /tmp/k.db --repo $TMPREPO --port 7700`
4. Open `http://localhost:7700`. Create a project (via curl POST /api/projects + repo_path = $TMPREPO) + a lahan via UI. Dispatch with mock agent batch=2.
5. Both attempts complete. Briefly: tab badges show pulsing pending dot. Within ~2s: both tabs show **red** dot (because `smoke-fail` exits non-zero; passing `smoke-pass` doesn't save it).
6. Click an attempt tab. The **Verification** section above the SSE log shows status "failed" + the aggregated block output:
   ```
   === smoke-pass (passed, 5ms) ===

   === smoke-fail (failed, 4ms) ===
   ```
7. Edit `.kulisawit/sortir.toml` — remove the failing check. Dispatch a fresh batch. Tabs eventually show **green** dot.
8. Delete `.kulisawit/sortir.toml`. Dispatch again. Tabs show **gray** dot. Verification section text: "no .kulisawit/sortir.toml found".
9. `cargo test --workspace --features kulisawit-server/embed-ui` ≥133 passed; `cd ui && npm test` ≥35 passed; `cargo clippy --workspace --all-targets --features kulisawit-server/embed-ui -- -D warnings` clean.
